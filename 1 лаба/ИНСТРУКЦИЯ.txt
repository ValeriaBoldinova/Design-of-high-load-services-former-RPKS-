========================================================

ВАЖНО: ПОЧЕМУ НЕ ПРОПАДЁТ СТАРАЯ БАЗА ДАННЫХ
------------------------------------------------------
Новый контейнер использует ПОРТ 5433 (не 5432).
Твоя старая база работает на порту 5432 — они НИКАК не
пересекаются. Также используется отдельный Docker-том
с уникальным именем "lab1_pgdata". Старый том и старый
контейнер не трогаются вообще.

Чтобы убедиться, что старое соединение в DBeaver цело:
  - порт: 5432 (или тот, что был)
  - контейнер: у старого своё имя, не "postgres_lab1"
  - том: у старого своё имя, не "lab1_pgdata"


СТРУКТУРА ФАЙЛОВ ПРОЕКТА
------------------------------------------------------
artemka/

ПРЕДВАРИТЕЛЬНЫЕ ТРЕБОВАНИЯ
------------------------------------------------------
1. Docker Desktop установлен и запущен
2. Терминал открыт в папке проекта:
     cd /path/to/artemka
   (или открыть папку artemka в терминале)
3. DBeaver установлен


ШАГИ ЗАПУСКА
------------------------------------------------------

ШАГ 1. Запустить контейнер с PostgreSQL
----------------------------------------
Открой терминал в папке "artemka" и выполни:

  docker-compose up -d

Что происходит:
  - Скачивается образ postgres:15 (если ещё нет)
  - Создаётся контейнер "postgres_lab1"
  - PostgreSQL стартует на порту 5433 (хост) → 5432 (контейнер)
  - Автоматически выполняются скрипты из папки sql/ в порядке:
      01_create_staging.sql  → создаёт таблицу mock_data
      02_load_data.sql       → загружает все 10 CSV
      03_ddl_schema.sql      → создаёт схему снежинки
      04_dml_fill.sql        → заполняет таблицы фактов/измерений

Инициализация занимает примерно 30–60 секунд.

Проверить, что контейнер запущен:
  docker ps

Посмотреть логи (и убедиться, что скрипты отработали):
  docker logs postgres_lab1


ШАГ 2. Подключиться через DBeaver (НОВАЯ база)
-----------------------------------------------
Создай НОВОЕ соединение в DBeaver:

  Тип базы:  PostgreSQL
  Host:      localhost
  Port:      5433          ← ВАЖНО: не 5432, а 5433
  Database:  lab1_snowflake
  Username:  lab1_user
  Password:  lab1_pass

Клик "Test Connection" → должно показать "Connected".

СТАРОЕ соединение в DBeaver НЕ ТРОГАЙ — оно остаётся на
своём порту и работает независимо.


ШАГ 3. Проверить результат
----------------------------
В DBeaver подключись к lab1_snowflake и выполни запросы
для проверки:

-- Количество строк в staging-таблице (должно быть ~10 000):
SELECT COUNT(*) FROM mock_data;

-- Проверка таблиц схемы снежинки:
SELECT COUNT(*) FROM fact_sales;
SELECT COUNT(*) FROM dim_customer;
SELECT COUNT(*) FROM dim_product;
SELECT COUNT(*) FROM dim_seller;
SELECT COUNT(*) FROM dim_store;
SELECT COUNT(*) FROM dim_date;
SELECT COUNT(*) FROM dim_country;
SELECT COUNT(*) FROM dim_product_category;
SELECT COUNT(*) FROM dim_pet_category;
SELECT COUNT(*) FROM dim_brand;
SELECT COUNT(*) FROM dim_supplier;
SELECT COUNT(*) FROM dim_pet;

-- Пример аналитического запроса (продажи по категориям):
SELECT
    dpc.category_name,
    SUM(fs.sale_total_price) AS total_revenue,
    COUNT(*)                 AS sales_count
FROM fact_sales fs
JOIN dim_product dp   ON dp.product_id   = fs.product_id
JOIN dim_product_category dpc ON dpc.category_id = dp.category_id
GROUP BY dpc.category_name
ORDER BY total_revenue DESC;


СХЕМА "СНЕЖИНКА" — ОПИСАНИЕ ТАБЛИЦ
------------------------------------------------------

ТАБЛИЦА ФАКТОВ:
  fact_sales          — продажи (центральная таблица)

ТАБЛИЦЫ ИЗМЕРЕНИЙ (уровень 1):
  dim_customer        — покупатели
  dim_seller          — продавцы
  dim_product         — товары
  dim_store           — магазины
  dim_date            — даты продаж

ПОД-ИЗМЕРЕНИЯ (уровень 2, делают схему "снежинкой"):
  dim_country         — страны (используется в customer, seller, store, supplier)
  dim_product_category — категории товаров
  dim_brand           — бренды товаров
  dim_supplier        — поставщики товаров
  dim_pet             — питомцы (связаны с покупателем)

ПОД-ИЗМЕРЕНИЕ (уровень 3):
  dim_pet_category    — категории питомцев (кошки/собаки/etc)

Цепочки нормализации (почему это "снежинка", а не "звезда"):
  fact_sales → dim_product → dim_product_category → dim_pet_category
  fact_sales → dim_customer → dim_country
  fact_sales → dim_product → dim_supplier → dim_country


УПРАВЛЕНИЕ КОНТЕЙНЕРОМ
------------------------------------------------------

Остановить (данные сохраняются):
  docker-compose stop

Запустить снова:
  docker-compose start

Остановить и удалить контейнер (данные в томе сохраняются):
  docker-compose down

Полный сброс (УДАЛИТ ВСЕ ДАННЫЕ ЛАБЫ, пересоздаст с нуля):
  docker-compose down -v
  docker-compose up -d


РЕШЕНИЕ ПРОБЛЕМ
------------------------------------------------------

Проблема: "port 5433 is already allocated"
Решение:  Порт 5433 занят другим процессом.
  docker ps  → найди, какой контейнер использует 5433
  Или измени порт в docker-compose.yml: "5434:5432"
  и в DBeaver тоже подключайся на 5434.

Проблема: "Connection refused" в DBeaver
Решение:  Контейнер ещё стартует или упал.
  docker ps          → проверь статус (должен быть "Up")
  docker logs postgres_lab1  → посмотри ошибки

Проблема: mock_data пустая / данные не загрузились
Решение:  Инициализационные скрипты запускаются ТОЛЬКО
  при первом старте (когда том пустой). Если том уже
  существовал — скрипты пропускаются.
  Сброс:
    docker-compose down -v
    docker-compose up -d

Проблема: "table mock_data does not exist" и т.п.
Решение:  Аналогично — полный сброс с -v (см. выше).

Проблема: Старая база в DBeaver "не существует"
Причина:  Скорее всего, был остановлен или удалён старый
  контейнер. Найди его:
    docker ps -a   → покажет все контейнеры (включая stopped)
  Если контейнер в статусе "Exited":
    docker start <имя_старого_контейнера>
  После запуска старый DBeaver-коннект снова заработает.
